# 氛围编程？编毛咧！

  <img src="/assets/creation/gg-a-7-p-1.png"/>
</div>

这个标题是用来吸引眼球的！原标题应该是《从氛围编程到超级应用》。

实际上，我认为氛围编程还会再流行一阵子，甚至在某个时间点会成为软件开发的主流。\
还在2025年初时，AI的代码还差强人意，时不时在一些简单的地方掉链子。\
进入2026年后，AI的进步巨大，很快就会把绝大部分开发者踢出局，也会成为众多软件公司的掘墓人。\
前些日子的SAAS股价大跌，预示着淘汰赛已经开始了。

然而，正如标题所要表达的，我认为氛围编程也只是一个过渡阶段的产物。

在软件开发里，我们曾多次重复这样的模式：
1. 起初，手工编写大量机械雷同的代码，质量完全取决于开发者的水平；
2. 然后，出现了好多高效的工具和成熟的库包，使得高质量的代码生成变得更容易；
3. 最后，随着技术的进步，这样的代码生成也渐渐消失了；

氛围编程的产物，那些应用，类比上文中的代码。在迎来一个全面开花的高潮后，也会渐渐的/迅速的消失。

看看我们每时每刻使用的应用，\
要打开这个娱乐，使用那个购物，在某些个应用上工作，在其它几个社交，还有健身，理财 ... 氛围编程的流行，会带来越来越多雷同的应用 ... \
假如，有一个聪明能干的“管家”在身边，凡事你只需指示它去完成，甚至它可能在你想到前已经处理完毕，规划妥当了。\
那么，谁还要装这个，用那个，在块小玻璃上点来戳去的呢？！

在我的设想里，取而代之的以AI为核心驱动的，功能极其丰富的，跨行业跨领域的超级应用。\
它具备以下特征：
### ・ AI驱动
这是核心——由一个存在于服务对象本地的AI驱动。\
这个AI负责收集信息，查询知识库，使用工具，分析处理当前任务和未来规划。\
它会和其它的私有AI，公共AI，在网络间保持着沟通联系，并且互相学习，自动（自我？）进化升级 ...

### ・ 静态知识
<a href="/creation/detail?id=gg-a-5" target="_blank">另篇文章</a>里聊过，静态的知识应该和推理系统分离的独立系统，是无须训练的，持续积累的，无论私有的还是世界的，全部供给推理系统分析使用的。

### ・ 使用与制造工具
既然是AI，理所当然的应该有使用和制造工具的能力。\
也就是说，已经固化的事物，不该每次都从头推理判断思考问题，而是和人类一样，使用工具来解决——也就是上面提到的各种应用。\
有望实现这个目标的种种规范概念协议，最早是Function Call，后来的MCP，然后Skill，最近又蹦出来Contract ... 实在很是需要一个通用的业界标准。

同样和人类一样，除了使用工具外，还要具有发明创造工具和持续改进工具的能力。\
这里就类似上面的氛围编程，只是主角从人类变成AI，AI自己把相对不变的处理逻辑，以应用或是什么形式固化下来，供自己使用。\
从这个意义上讲，应用没有消失，而是变成了AI使用的“工具”。

举例来说，
| 情景 | 机制 |
| ------------- | ------------- |
| 每个人都有个电子管家在身边照顾我们的生活起居 | 存在于服务对象本地的AI |
| 譬如，它会每天早晨煮上一杯健康，美味，合我们口味的咖啡 | 健康美味的配比以及煮咖啡的方法来自于世界知识；合口味则是私有的知识 |
| 有天AI突然，制造/订购了台咖啡机 | 制造/使用工具 |
| 有天AI突然，从（AI驱动的）某餐饮平台订购了咖啡 | 与其它AI交流协作 |


用这个彷佛一个古旧的科幻小说情节，来尝试说明这个超级应用的核心机制。\
这个超级应用也应该是应用的最终形态。\
再往后，AI也许会以某种，我们无法想象的方式，全面接手了。
